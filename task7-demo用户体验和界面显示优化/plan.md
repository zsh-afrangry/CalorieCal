# Task7：/#/demo 用户体验和界面显示优化

## 本 task 要完成的内容

Task7 重点关注 /#/demo 的用户体验和界面显示，面向商场液晶屏、外场演示和路人快速理解。它不优先新增算法能力，而是把 Task4-Task6 已经形成的运动量、动作证据、RGB-D 状态、强度、kcal 和置信度，以更清晰、更稳定、更适合大屏的方式展示出来。

## 预期结果

1. /#/demo 第一眼能看懂：当前人在动什么、强度如何、估算消耗如何、系统是否可靠。
2. 界面适合商场液晶屏，文字大、层级清晰、无需操作按钮，不依赖用户阅读复杂说明。
3. 摄像头画面、骨架、当前动作、强度、kcal、质量提示之间有明确视觉优先级。
4. depth / RGB-D 状态只作为可读提示出现，不展示过多调试细节。
5. 当识别质量下降时，页面能自然提示“请正对摄像头 / 请进入画面 / 估算置信度下降”。
6. 后续可扩展排行榜、挑战模式、历史累计或活动引导，但 Task7 优先保证基础展示体验。
7. 人体可视化比当前细线骨架更清楚：躯干、大腿、小腿等部位在显示上更像身体结构，而不是只有一条线。

## 可能的实现方案和技术栈

1. 前端：继续使用 Vue 3 + TypeScript + 当前 CSS。
2. 页面：只改 /#/demo，不大改 /#/pose 和 /#/gesture。
3. 展示：左侧保持摄像头和骨架为核心，右侧或底部展示少量关键指标。
4. 文案：面向路人，避免工程调试词；调试信息保留在 /#/pose。
5. 响应式：优先考虑 16:9 横屏液晶屏，同时保证普通浏览器窗口不溢出。
6. 人体结构显示：继续基于 MediaPipe Pose 现有关键点，使用更粗的肢体线、半透明躯干面、关节节点和左右肢体颜色区分增强可读性。

## 边界

1. Task7 不负责新增动作识别算法。
2. Task7 不负责解决 SAM 3D Body、registration 或侧身算法问题。
3. Task7 不追求商业化完整活动页，只做演示版界面体验优化。
4. Task7 不在页面放复杂操作按钮，/#/demo 应尽量自动运行。
5. Task7 只做“显示更详细”，不做“识别更详细”；DensePose、人体分割、SAM 3D Body 或人体 mesh 等识别/模型升级放到 Task8。

## 动作识别扩展判断

1. 需要引入 Task9：基础动作和组合动作新增。原因是“尽可能增加动作识别”需要动作定义、阈值、置信度、状态机、测试动作清单和逐个调参，已经超出 Task7 的展示体验范围。
2. Task7 只负责让 /#/demo 能承接更多动作结果：当前基本动作、当前组合动作、强度、kcal、质量提示和动作解释的展示区域要可扩展，不把某个动作写死成唯一主角。
3. Task8 负责“识别能力是否更细”，例如人体分割、DensePose、RGB-D 区域采样、人体 mesh；Task9 负责“在现有或增强后的识别能力上增加动作库”。
4. Task9 应优先增加工位和外场都容易测试的动作，再扩展到全身动作；每个动作都需要记录误报、漏报、阈值依据和验收结果。

## Task9 建议动作范围

### 基础动作候选

1. 上肢：左手上抬、右手上抬、双臂上抬、手臂回落、手向前移动、手向后移动、手水平摆动、手臂伸展、手臂弯曲。
2. 下肢：双脚打开、双脚合拢、左抬腿、右抬腿、膝盖上抬、下蹲、起身。
3. 躯干：身体前倾、身体后仰、身体左倾、身体右倾、身体左转、身体右转。
4. 头部 / 面部：点头、摇头、左转头、右转头；这些更适合低强度活动提示，不直接作为高 kcal 动作主依据。

### 组合动作候选

1. 静息：检测到人体 + 运动量低 + 持续稳定，用于避免演示页在站立不动时显示“未识别组合动作”。
2. 挥手：手水平摆动 + 持续周期 + 手部可见。
3. 出拳：手向前移动 + 手臂伸展 + 回收，可结合 RGB-D 前后方向证据。
4. 开合跳：双脚打开 / 合拢 + 双臂上抬 / 回落 + 周期状态机。
5. 高抬腿：左 / 右膝盖上抬 + 躯干稳定 + 左右交替周期。
6. 深蹲：下蹲 + 起身 + 膝髋高度变化 + 躯干参考稳定。
7. 侧摆臂 / 原地摆臂：左右手或双臂持续摆动 + 中低强度活动累计。

### 优先级建议

1. P0：先做基础动作库扩展和显示，不急于让所有组合动作计数。
2. P1：优先做挥手、出拳、开合跳、高抬腿、深蹲五个组合动作，因为它们覆盖上肢、前后方向、全身周期和下肢动作。
3. P2：补充头部 / 躯干动作，主要用于演示丰富度和质量反馈，不作为主要 kcal 估算依据。
