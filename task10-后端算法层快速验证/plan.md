# Task10：后端算法层快速验证

## 1. Task10 的目标

Task10 面向 ToC 商业化产品前期验证，不做学术研究型的大模型探索，也不继续堆叠更多识别点。目标是在现有前端已经能稳定获取人体点位、手部信息、segmentation mask 和 RGB-D depth 采样结果的基础上，快速搭建一个“后端算法层”，验证如何更有效地利用点位序列、depth 证据、动作样本和时序特征，尽快获得可展示、可量化、可迭代的动作识别效果。

这里的“后端”包含算法层：可以是 Python 服务，也可以先是独立脚本 / 本地原型；重点是把前端点位或离线样本转成稳定的动作结果，而不是让前端继续承担越来越多阈值和状态机细节。

Task10 的核心不是“找一个神秘万能模型”，而是建立一条可复用链路：

公开视频 / 图片数据集 / 自录视频  
→ 统一提取人体姿态关键点  
→ 清洗、归一化、标注动作阶段  
→ 生成动作样本资产  
→ 从样本资产生成动作参数  
→ 接入轻量识别算法  
→ 本地视频 / 摄像头实时验证  
→ 输出量化指标  
→ 决定是否接入 Vue 或后端 API

## 2. 商业化前期约束

1. 不追求论文级 SOTA，不引入训练成本高、部署链路重、解释困难的模型作为第一步。
2. 不优先继续 Whole-Body 133 点、DensePose、SAM 3D Body 或完整人体 mesh；现有点位对当前动作库已经基本足够。
3. 优先选择容易复刻、容易调参、能在用户实测反馈中快速迭代的技术。
4. 每一轮验证必须能量化：响应速度、识别延迟、识别稳定性、动作准确率、误报 / 漏报场景。
5. 可视化成果要服务外场演示：用户做动作后，屏幕能更快、更稳定、更可信地反馈。
6. 自录数据集是可接受且重要的路线；公开数据集用于起步和补充，自录样本用于贴近真实产品场景。

## 3. 第一轮动作范围

第一轮动作库保留 6 类：

1. 静息
2. 挥手
3. 出拳
4. 深蹲
5. 高抬腿
6. 开合跳

但实现优先级分三步：

1. 第一轮重点闭合：深蹲、开合跳。
2. 第二轮扩展：挥手、出拳。
3. 第三轮扩展：高抬腿。

原因是深蹲和开合跳最适合从公开数据集和自录样本生成参数资产，也最容易量化“是否识别正确”和“是否完成一次动作”。挥手、出拳依赖更多混淆抑制；高抬腿公开样本不够理想，更适合先用自录数据和规则模板校准。

## 4. 数据路线

Task10 使用“公开数据集 + 自录数据集”双路线。

### 4.1 公开数据集

公开数据集主要用于快速获得初始样本、动作阶段参考和参数分布参考。

1. 深蹲 / 开合跳：优先 PoseRAC / RepCount-pose、Penn Action。
2. 挥手 / 出拳：优先 NTU RGB+D、UCF101。
3. 高抬腿：公开数据不理想，暂不依赖公开数据作为主来源。

公开数据集需要记录：

1. 数据集名称。
2. 数据来源和下载方式。
3. 包含哪些动作。
4. 是视频、图片、2D 姿态、3D skeleton，还是已有关键点标注。
5. 许可证和商业使用风险。
6. 是否进入最终产品资产，还是只用于内部研究参考。

### 4.2 自录数据集

自录数据集是第一轮参数校准的核心来源，因为它更贴近真实摄像头、真实用户、真实房间和当前产品使用方式。

建议第一轮采集量：

1. 每个动作 10~30 条正样本。
2. 每个动作 10~20 条负样本 / 混淆样本。
3. 每条视频 3~8 秒，包含动作开始、完整动作、动作结束。
4. 图片样本用于静态阶段，例如深蹲最低点、开合跳打开态、出拳伸展态。

视频采集要求：

1. 摄像头固定，不频繁移动。
2. 人体完整入镜，头、手、脚不要出画面。
3. 第一轮以正面视角为主，后续再增加侧面 / 斜侧面。
4. 光线稳定，背景尽量减少遮挡。
5. 文件名包含动作名、样本类型、编号、视角，例如 `jumping_jack_positive_front_001.mp4`。

图片采集要求：

1. 图片适合采集静态姿态阶段，不适合单独表达完整动作。
2. 每个动作阶段可拍 20~50 张图片。
3. 开合跳阶段：`closed`、`open`。
4. 深蹲阶段：`stand`、`down`。
5. 高抬腿阶段：`left_knee_up`、`right_knee_up`。
6. 挥手阶段：`hand_left`、`hand_right`、`hand_high`。
7. 出拳阶段：`guard`、`extend`、`retract`。

5. `quality`：good / usable / rejected。

## 6. 图片 / 视频到姿态关键点

Task10 第一版统一使用身体主骨架，不继续依赖更多细节点位。

姿态提取选择：

1. 优先使用与当前前端一致的 MediaPipe Pose 33。
2. MMPose COCO-17 可作为对照。
3. Whole-Body 133 点保留为可选输入，但第一轮动作识别只使用身体主骨架。

核心关键点：

1. 鼻子。
2. 左右肩。
3. 左右肘。
4. 左右腕。
5. 左右髋。
6. 左右膝。
7. 左右踝。
8. 脚跟和脚尖。

图片转姿态：

1. 输入：单张图片。
2. 输出：单帧 pose JSON。
3. 保存字段：`image_path`、`action`、`stage_label`、`landmarks`、`visibility`、`image_width`、`image_height`、`extractor`、`created_at`。
4. 用途：生成静态阶段模板。

视频转姿态：

1. 输入：mp4 / webcam record。
2. 输出：多帧 pose sequence JSON / JSONL。
3. 每帧保存：`timestamp`、`frame_index`、`landmarks`、`visibility`、`extractor_fps`、`pose_quality`。
4. 用途：生成动作时序模板。

姿态质量检查：

1. 关键点可见率过低时标记低质量。
2. 手腕、髋、膝、踝缺失时，对上肢 / 下肢动作分别降权。
3. 人体框过小、身体出画面、多人入镜时标记异常。
4. 不合格样本不进入参数生成，只进入问题样本列表。

## 7. 姿态关键点到动作样本资产

姿态样本不能直接用于动作识别，必须先转成动作资产。

### 7.1 坐标归一化

归一化目标是让不同身高、不同远近、不同画面尺寸下的动作可比较。

第一版归一化方式：

1. 以髋中心作为人体中心。
2. 以肩宽、髋宽或躯干长度作为尺度。
3. 使用相对坐标，而不是原始像素坐标。
4. 对左右方向、高度、距离做统一表达。

### 7.2 Pose embedding

把原始关键点转成动作特征：

1. 静态特征：关节角度、左右距离、手腕相对肩 / 髋位置、膝盖相对髋位置、脚距 / 肩宽。
2. 时序特征：速度、加速度、方向变化、动作幅度、周期时长。
3. depth 特征：手腕相对肩 / 髋的前后位移，主要用于出拳。
4. 质量特征：关键点置信度、可见率、样本质量分。

### 7.3 动作阶段

第一版阶段定义：

1. 开合跳：`closed`、`opening`、`open`、`closing`。
2. 深蹲：`stand`、`downing`、`down`、`rising`。
3. 挥手：`left_swing`、`right_swing`、`turning`。
4. 出拳：`guard`、`extending`、`extended`、`retracting`。
5. 高抬腿：`left_up`、`left_down`、`right_up`、`right_down`。

### 7.4 动作样本资产

每个动作至少包含：

1. 正样本：标准动作和可接受变体。
2. 负样本：静息、随机小动作、无关动作。
3. 混淆样本：容易误判为该动作的相邻动作。
4. 阶段标签：动作内部阶段。
5. 关键曲线：例如髋部高度、膝角度、脚距、腕部高度、腕部横向位移、手腕 depth。
6. 样本质量分：用于排除不可靠样本。

建议资产目录：

```text
backend/action_engine/assets/
  pose_samples/
  templates/
  curves/
  negative_samples/
  reports/
```

## 8. 动作样本资产到动作参数

每个动作都必须沉淀为一份可版本化的 action spec。它可以是 JSON / YAML / Python dict，第一版不追求优雅，优先保证能调、能看、能复现。

建议目录：

```text
backend/action_engine/action_specs/
```

第一批 spec：

```text
rest
wave
punch
squat
high_knee
jumping_jack
```

每个 action spec 至少包含：

1. `action_name`：动作名。
2. `action_type`：static / transition / cyclic。
3. `required_landmarks`：必须可见的关键点。
4. `features`：使用哪些特征。
5. `feature_weights`：特征权重。
6. `enter_threshold`：进入阈值。
7. `exit_threshold`：退出阈值。
8. `min_hold_ms`：最小保持时间。
9. `cooldown_ms`：冷却时间。
10. `templates`：正样本模板。
11. `negative_templates`：负样本模板。
12. `confusion_templates`：混淆样本模板。
13. `quality_gates`：质量门控。
14. `explain_rules`：可解释原因。

参数生成方式：

1. 从样本统计每个阶段的关节角度范围。
2. 从样本统计每个阶段的相对高度范围。
3. 从样本统计左右距离、脚距、手距范围。
4. 从正样本和负样本的特征分布差异中生成阈值。
5. 用分位数生成初始阈值，不凭感觉手写死阈值。
6. 从时序样本统计动作周期、动作幅度、速度峰值、方向变化次数。
7. 生成状态机的阶段进入 / 退出条件。
8. 生成 KNN / DTW / 模板相似度阈值。

第一轮动作参数重点：

1. 开合跳：手腕高度、手距、脚距、脚距 / 肩宽、`closed→open→closed` 周期。
2. 深蹲：髋部下降幅度、膝髋角度、躯干倾斜、`stand→down→stand` 周期。
3. 挥手：手腕高度、横向摆动幅度、换向次数、身体整体运动量限制。
4. 出拳：手臂伸展、手腕 depth 前移、回收、与挥手互斥。
5. 高抬腿：膝盖相对髋部高度、左右交替、身体中心稳定性。

## 9. 动作参数到识别算法

第一版算法由三层组成：

1. 规则状态机。
2. KNN / 模板相似度。
3. DTW / 时序模板匹配。

### 9.1 Compact frame JSON

输入字段：

1. `timestamp`
2. `pose33`
3. `handSummary`
4. `depthSamples`
5. `maskQuality`
6. `frontendFps`

输出字段：

1. `basicActions`
2. `compositeAction`
3. `confidence`
4. `latencyMs`
5. `processingFps`
6. `evidence`
7. `debugReason`

### 9.2 Python 本地原型

第一阶段优先本地视频 / 窗口验证，不急着接 Vue。

原型需要：

1. 支持离线视频输入。
2. 支持摄像头实时输入。
3. 维护 1 秒和 3 秒 rolling buffer。
4. 输出每帧动作结果 JSON / JSONL。
5. 可视化骨架、动作名、阶段、置信度、失败原因。

### 9.3 规则状态机

规则状态机负责动作阶段和计数：

1. 从 action spec 读取阈值和阶段定义。
2. 每个动作包含 enter、hold、stage transition、exit、cooldown。
3. 输出当前动作阶段和失败原因。
4. 避免把参数散落在代码里。

### 9.4 KNN / 模板相似度

KNN / 模板相似度负责回答“当前姿态像不像某个动作阶段”：

1. 使用 pose embedding 与正样本 / 负样本比较。
2. 输出动作相似度分数。
3. 记录最近邻样本 ID，方便追踪“为什么像这个动作”。
4. 与规则状态机分数融合。

### 9.5 DTW / 时序模板匹配

DTW 负责回答“一段动作序列像不像完整动作”：

1. 使用 1~3 秒 rolling buffer。
2. 对关键曲线做时序对齐。
3. 用于深蹲、开合跳、高抬腿等周期动作。
4. 只保留能明显降低误报 / 漏报的模板逻辑。

### 9.6 分数融合

第一版融合公式可以保持简单：

```text
final_score = rule_score + template_score + quality_score - confusion_penalty
```

融合原则：

1. 进入动作需要高阈值。
2. 退出动作使用低阈值，形成滞回。
3. 静息状态要有保护，避免轻微抖动频繁触发动作。
4. 同一时间多个动作冲突时，按 spec 中的互斥规则处理。

## 10. 本地验证和可视化

### 10.1 离线视频验证

输入公开视频 / 自录视频，输出：

1. 带骨架、动作名、阶段、置信度、失败原因的可视化视频。
2. 每帧识别结果 JSONL。
3. 测试汇总 JSON。

### 10.2 图片阶段验证

输入自录图片或公开图片，输出：

1. 姿态点可视化。
2. 阶段分类结果。
3. 命中特征。
4. 失败原因。

### 10.3 摄像头实时验证

本地窗口实时显示：

1. 人体骨架。
2. 动作名。
3. 当前阶段。
4. 置信度。
5. 关键证据。
6. 失败原因。

示例证据：

1. `脚距达到 1.8x 肩宽`
2. `髋部下降 0.32 torso`
3. `手腕高于肩部 0.24 torso`
4. `未完成 closed→open→closed`

## 11. 量化指标

速度目标：

1. 后端算法处理耗时：目标 < 10ms / 帧。
2. 端到端响应延迟：目标 < 150ms，优秀 < 100ms。
3. 动作开始后 300~600ms 内给出可见反馈。

准确率目标：

1. 深蹲正例识别率：第一轮 > 80%。
2. 开合跳正例识别率：第一轮 > 80%。
3. 静息 / 混淆动作误报率：第一轮 < 15%。
4. 单个动作持续 10 秒时异常跳变 < 2 次。

可解释目标：

1. 每个动作结果必须带 1~3 条证据。
2. 每个拒绝识别结果必须带失败原因。
3. 每轮调参必须能追溯是哪些样本、特征或阈值发生变化。

## 12. 接入决策

Task10 不默认接入 Vue。接入前先判断后端算法层是否真的带来收益。

接入条件：

1. 响应速度、稳定性或准确率明显优于当前前端规则。
2. 至少深蹲和开合跳有稳定可视化效果。
3. 误报和漏报能被记录并继续迭代。
4. 动作结果能用 compact JSON 输出。

接入形式：

1. 如果只是验证和演示，继续 Python 本地窗口最快。
2. 如果要产品化体验，封装成 Python API / WebSocket。
3. Vue 负责展示，算法层负责识别。
4. 如果后续需要部署，action spec、模板和轻量模型需要独立版本化。

## 13. Task10 边界

1. Task10 不负责继续扩展前端 UI。
2. Task10 不负责继续研究人体识别点位是否更多。
3. Task10 不以训练大模型为目标。
4. Task10 不追求一次性解决全部动作。
5. Task10 只追求最快得到可展示、可量化、可迭代的后端算法成果。

1. 数据集目录和样本标注表。
