# Task2：多运动证据评分

## 本 task 要完成的内容

把 `/#/gesture` 中“单一当前动作”的判断方式，改成“多运动证据评分”。重点解决 Task1 中挥手被“握拳 / 手前移 / 手后移 / 出拳”等规则抢占，导致挥手计数中断或漏计的问题。

本 task 的重点不是新增更多手势名称，而是让系统同时观察多个运动证据：

```text
横向摆动证据
前后移动证据
手指弯曲/握拳证据
挥手周期证据
关键点稳定性证据
```

每个证据可以有独立的 0-100 评分，不要求所有动作共享 100 分。动作标签只是辅助展示，热量估算应优先依赖运动时间、频率、幅度、强度和置信度。

## 预期结果

1. 页面不再只依赖 `currentGesture` 作为唯一识别结果。
2. 挥手计数使用更宽松的 `waveEvidence` 或 `waveCounterSignal`，允许短暂出现“手前移 / 手后移 / 握拳”但不中断整个挥手过程。
3. 展示层可以显示多个候选状态及各自置信度，例如“挥手 72、手前移 45、握拳 38”。
4. 计数逻辑和展示标签拆分：展示标签说明当前画面像什么，计数标签服务于周期统计和 kcal 估算。
5. 置信度原因更可解释，能说明漏计、低置信度或规则冲突来自哪里。

## 可能的实现方案和技术栈

1. 前端：继续使用 Vite + Vue 3 + TypeScript。
2. 视觉识别：继续使用 MediaPipe Hand Landmarker。
3. 算法：规则算法 + 时间窗口 + 多证据独立评分。
4. 数据结构：新增 `motionEvidence` 或 `gestureCandidates`，包含横向摆动、前后移动、手指弯曲、周期稳定性等评分。
5. 调试：在页面调试面板展示每项证据分数、命中规则、被抑制规则和置信度原因。

## 边界

1. 不把 `/gesture` 扩展成“手势大全”。
2. 不要求快速挥手、严重残影、遮挡和光照差全部解决。
3. 暂不引入 LSTM、Transformer、ST-GCN 等模型。
4. kcal 仍以“演示可解释”为主，不追求贴近真实代谢消耗。
