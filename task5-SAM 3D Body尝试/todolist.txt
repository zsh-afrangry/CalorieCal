# Task5 SAM 3D Body 尝试 — 实施清单

> 状态：待开始。  
> 目标：评估 Meta SAM 3D Body / 3DB 是否值得进入本项目，不默认替换当前 MediaPipe 主链路。  
> 原则：先研究和验证，再决定是否接入；优先关注它是否能改善深度、遮挡、身体比例、3D 姿态和热量估算置信度。  

---

## Phase 1: 范围确认和资料核对

- [ ] 1.1 核对官方资料
      → 确认论文、官方仓库、checkpoint、许可证、硬件要求和推理流程
      → 输出: 是否具备本地尝试条件

- [ ] 1.2 明确 Task5 不替换主链路
      → MediaPipe Pose / Hand 仍作为实时前端主链路
      → SAM 3D Body 只作为 3D 人体网格恢复可行性评估

- [ ] 1.3 明确评估指标
      → 运行成本、实时性、模型体积、安装复杂度、输出质量、对 kcal 估算是否有增益

## Phase 2: 最小离线推理尝试

- [ ] 2.1 准备 Python 环境方案
      → 优先按官方仓库说明创建独立环境
      → 不污染 frontend 依赖

- [ ] 2.2 准备测试输入
      → 使用单张人体图片或短视频抽帧
      → 保留输入样例和输出路径说明

- [ ] 2.3 跑通一次官方推理
      → 输出人体 mesh / 可视化结果 / 关键日志
      → 如果硬件或 checkpoint 受限，记录阻塞原因

## Phase 3: 与当前 MediaPipe 链路对比

- [ ] 3.1 对比输出能力
      → MediaPipe: 关键点、实时性、浏览器端可用
      → SAM 3D Body: 3D mesh、身体表面、深度和比例信息

- [ ] 3.2 对比对热量估算的实际价值
      → 判断是否能改善位移、幅度、前后方向动作、遮挡和置信度
      → 不只看视觉效果好不好看

- [ ] 3.3 记录部署成本
      → 是否需要 GPU、是否适合后端、是否能实时、是否能在商场演示场景使用

## Phase 4: 结论和后续方案

- [ ] 4.1 给出接入判断
      → 立即使用 / 只做离线分析 / 暂缓

- [ ] 4.2 如果暂缓，记录原因
      → 例如模型过重、实时性不足、安装复杂、对 kcal 增益不明显

- [ ] 4.3 如果继续，拆分后续任务
      → 后端推理服务、前端 mesh 可视化、与 MediaPipe 特征融合、置信度融合

- [ ] 4.4 构建或脚本验证
      → 如修改前端，执行 npm.cmd --prefix frontend run build
      → 如新增 Python 脚本，执行对应最小运行或语法检查

## 待商讨记录区

### Task5 的定位

Task5 是研究型任务，不是 MVP 必须项。它的价值必须落到运动时间、频率、幅度、强度、估算消耗和置信度，而不是单纯展示 3D 人体模型。

### 可能的结论

如果 SAM 3D Body 对实时性和部署成本不友好，短期可以继续使用 MediaPipe 主链路，把 SAM 3D Body 留作离线分析或论文展示方向。
