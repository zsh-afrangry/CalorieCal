# Task8 更详细人体识别方案研究 — 实施清单

> 状态：已开始，先评估当前已有轻量方向。  
> 目标：研究比 MediaPipe Pose 33 点更细的人体识别、人体表面、Whole-Body Pose 和 RGB-D 区域融合方案，判断是否能改善侧身、遮挡、躯干基准不稳定和空间幅度估算。  
> 原则：优先选择轻量、可实时、能接入现有 Pose/Hand/RGB-D 链路的方案；SAM 3D Body、重型 HMR 和完整 mesh 暂缓，不为了 3D 而 3D。

## Phase 2: 人体分割方案评估

- [x] 2.1 评估 MediaPipe segmentation mask
      → 能否提供人体轮廓
      → 能否辅助过滤 depth 背景
      → 是否适合浏览器实时运行
      → 是否能直接复用当前 MediaPipe Pose Landmarker 输出
      → 已确认: MediaPipe Pose Landmarker Web 支持 `outputSegmentationMasks` 选项，默认值为 false；开启后结果中可返回 `segmentationMasks`
      → 已确认: segmentation mask 表示像素属于检测到的人体的可能性，可用于人体轮廓、背景过滤和 depth 采样辅助
      → 已确认: 当前项目使用的 `@mediapipe/tasks-vision` 类型定义中也存在 `outputSegmentationMasks?: boolean` 和 `PoseLandmarkerResult.segmentationMasks`
      → 可行性判断: 可以优先在 /#/pose 或新增测试 tab 中开启，不需要先引入 Python 后端

- [x] 2.2 评估普通人体分割模型

      → 已确认: MediaPipe Image Segmenter Web 可作为独立人体 / 图像分割方案，但 `segment()` / `segmentForVideo()` 是同步调用，处理摄像头视频帧时会阻塞 UI 线程

      → 风险: 独立分割模型需要额外模型加载、额外推理时延和 mask 对齐逻辑；若再叠加 Pose/Hand/RGB-D，外场设备性能压力会增加

      → 用人体 mask 过滤背景 depth

- [x] 3.3 评估 depth 对动作识别的实际收益
      → 出拳: 前后方向是否更稳定
      → 手向前 / 手向后: 是否减少误报
      → 躯干基准: 是否减少侧身或遮挡造成的异常

      → 预期收益: 若手腕 depth 点落在人像 mask 内且 foregroundDepthMm 稳定，则后续可用于出拳、手向前 / 手向后；若肩髋中心命中稳定，则可用于躯干 depth 基准
      → 结论: mask + RGB-D 可以先作为动作识别接入前的质量门控和采样可信度反馈，不急于直接改 /#/demo
      → 边界: 本阶段只做测试页评估，不把结果接入 /#/demo 的动作识别状态机

## Phase 4: Whole-Body Pose Estimation 测评

- [x] 4.1 记录 Whole-Body Pose 关键点规模和价值
      → COCO-WholeBody 常见为 133 个关键点: 身体 17、脚 6、脸 68、双手 42
      → 价值: 比 MediaPipe Pose 33 点更细，尤其补足脚部、手部和面部密集关键点
      → 边界: 仍是 2D/关键点估计，不等于人体表面 mesh，也不直接提供肌肉或身体体积
      → 定位: Whole-Body Pose 仍属于关键点路线，优先级高于 DensePose / SAM 3D Body 这类更重的区域或 mesh 方案
      → 已确认: COCO-WholeBody / MMPose 方向适合评估“更多关键点是否能增强动作识别”，尤其是脚、手、脸这些当前 MediaPipe Pose/Hand 组合仍需手动拼接的部位
      → 已确认: Whole-Body Pose 的价值不是替代 segmentation mask 或 RGB-D，而是补足关键点密度；mask + depth 仍负责人体区域和前后距离可信度

- [x] 4.2 评估候选技术栈
      → OpenMMLab MMPose / RTMPose / RTMW
      → 推理后端: Python、ONNXRuntime、TensorRT、ncnn 或 MMDeploy
      → 判断是否适合当前 Windows 开发环境和未来外场设备
      → 已确认: MMPose 支持 whole-body pose estimation，RTMPose / RTMW 是主要候选
      → 已确认: RTMPose 强调实时多人体姿态估计，论文报告了较高实时性能；RTMW 面向 2D/3D whole-body，覆盖 body / hands / face / feet
      → 已确认: MMDeploy 支持将 MMPose 模型转换到 ONNXRuntime、TensorRT、ncnn、OpenVINO 等后端，可用于后续部署评估
      → 技术路线建议: 第一轮先做 Python + MMPose/RTMW 单帧或视频离线验证；第二轮再考虑 ONNXRuntime 或 TensorRT；暂不直接塞进 Vue 前端

- [x] 4.3 评估对本项目动作识别的收益
      → 脚部关键点: 双脚打开 / 合拢、高抬腿、开合跳、深蹲边界
      → 手部关键点: 挥手、出拳、握拳、手臂姿态
      → 面部关键点: 点头、摇头、转头质量反馈
      → 目标: 判断是否能增强脚部、手部和面部动作识别，而不是追求更复杂的人体建模
      → 风险: 133 点可能增加计算量、部署复杂度和前端显示拥挤
      → 预期收益: 脚部 6 点可能改善“双脚打开 / 合拢”和下肢动作边界；手部 42 点可减少单独 Hand Landmarker 与 Pose 拼接误差；脸部 68 点可增强头部动作和朝向质量反馈
      → 现实边界: 当前项目已有 MediaPipe Hand 和 Face 旧实现，Whole-Body Pose 的新增价值主要是统一模型和统一坐标，不一定天然比现有单项模型更准
      → 风险记录: 133 点会让显示层更拥挤，也可能引入较重 Python 环境、模型权重、检测框前处理和多后端部署复杂度

- [x] 4.4 判断是否进入实验 tab
      → 若实时性可接受，新增独立测试 tab，不影响 /#/demo
      → 若实时性一般，先做 Python 后端或离线视频测评
      → 若部署成本过高，只保留为研究记录
      → 结论: 值得后续进入独立实验链路，但不建议立即做 Vue 端实时接入
      → 建议新增后续任务: 先创建 Python 脚本或小服务，读取摄像头 / 视频帧并输出 133 点 JSON；验证 FPS、关键点稳定性和与现有 MediaPipe 关键点的差异
      → 实验 tab 条件: 当 Python/ONNX 链路能稳定输出 133 点 JSON 后，再新增 /#/wholebody-test 可视化，不影响 /#/demo
      → 暂不接入原因: 当前 mask + RGB-D 已经能解决一部分质量反馈问题；Whole-Body Pose 应先证明它对动作识别收益大于部署成本

## Phase 5: DensePose / 细粒度人体区域研究记录

- [x] 5.1 评估 DensePose
      → 是否能区分躯干、大腿、小腿、手臂等人体区域
      → 是否适合 Python 后端
      → 是否能服务运动幅度和置信度
      → 已确认: DensePose 的核心是把 RGB 图像中的人体像素映射到人体 3D 表面表示，属于 dense correspondence / body surface 路线，不是普通关键点检测
      → 已确认: DensePose-COCO 标注了 COCO 中约 50K 人体实例的图像到身体表面对应关系；Detectron2 中包含 DensePose 项目和模型 zoo，可作为 Python 后端研究入口
      → 对本项目的潜在价值: 可把人体像素划分到更细的人体表面区域，用于躯干、大腿、小腿、手臂等区域级运动量和 depth 区域过滤
      → 与 Whole-Body Pose 的区别: Whole-Body Pose 提供更多关键点，更适合动作边界；DensePose 提供人体表面区域，更适合区域 mask、身体表面参与度和展示/离线分析
      → 与 segmentation mask 的区别: segmentation mask 只告诉“是不是人体”，DensePose 可进一步告诉“人体的哪个表面区域”
      → 结论: DensePose 对“细粒度人体区域”有研究价值，但当前优先级低于 segmentation mask、RGB-D 区域采样和 Whole-Body Pose

- [x] 5.2 判断实时性和部署成本
      → 是否需要 GPU
      → 是否能在当前演示设备上运行
      → 是否只适合离线实验
      → 若成本过高，保留为后续研究，不进入当前实时链路
      → 已确认: DensePose 当前主要通过 Detectron2 / Python / PyTorch 链路使用，浏览器端直接实时接入成本较高
      → 已确认: Detectron2 官方 model zoo 支持加载配置和预训练权重，DensePose 也已整合到 Detectron2 项目中；但 Windows 本机安装、CUDA/PyTorch 版本、模型权重和推理速度都需要单独验证
      → 部署判断: 不建议直接接入 /#/demo；若要验证，先做 Python 离线图片 / 视频脚本，或独立后端服务输出区域 mask / IUV，再考虑 /#/densepose-test
      → 性能风险: 即使论文和模型报告过实时或多帧性能，实际外场设备还要叠加 Pose/Hand/RGB-D/前端展示，整体延迟可能不可接受
      → 维护风险: DensePose 输出 IUV / body surface 区域，后续还需要把区域结果映射到本项目的动作强度、幅度、频率和 cal 解释口径，工程工作量不小

- [x] 5.3 Phase 5 收尾结论
      → 当前不实现 DensePose，不新增前端 tab
      → 近期优先级: 继续沿用 segmentation mask + RGB-D 质量门控；Whole-Body Pose 若要推进，先做 Python/ONNX 133 点 JSON 验证
      → DensePose 后续用途: 作为离线分析、论文展示、人体表面区域参与度、肢体区域 depth 过滤和未来 3D/mesh 方向的过渡研究

## 附录：Phase 7 环境失败记录

### A.1 `CalorieCal` 环境失败记录

- 失败环境: conda env `CalorieCal`，Python 3.10
- 失败组合: torch 2.11.0+cu128 / CUDA 12.8 / numpy 1.26.4 / mmcv 2.1.0 / mmengine 0.10.7 / mmdet 3.2.0 / mmpose 1.3.2
- 失败现象: `MMPoseInferencer('wholebody')` 导入失败，出现 `ModuleNotFoundError: No module named 'mmcv._ext'`
- 失败原因判断: 当前安装链路实际缺少 OpenMMLab 编译扩展，类似 `mmcv-lite` 或无匹配 wheel 状态；MMPose 导入 transformer heads 时触发 `MultiScaleDeformableAttention` / `mmcv.ops`
- 处理结论: torch 2.11.0+cu128 太新，Windows 下难以获得匹配的 mmcv 预编译 wheel；不建议在该环境继续源码编译 mmcv

### A.2 `CalorieCalPose` NumPy 2.x 失败记录

- 失败环境: conda env `CalorieCalPose`，Python 3.10
- 失败组合: torch 2.1.2+cu121 / mmcv 2.1.0 / mmdet 3.2.0 / mmpose 1.3.2 / numpy 2.2.6
- 失败现象: import torch / mmcv / mmpose 时出现 `_ARRAY_API not found`；导入 `xtcocotools._mask` 时出现 `numpy.dtype size changed`
- 失败原因判断: torch 2.1.2、mmcv、xtcocotools 等二进制包按 NumPy 1.x 编译，不能直接运行在 NumPy 2.2.6 上
- 处理结论: 将 NumPy 固定到 `1.26.4` 并重装 `xtcocotools` 后，`mmcv._ext`、`xtcocotools._mask` 和 `MMPoseInferencer('wholebody')` 均验证通过
