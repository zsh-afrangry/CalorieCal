# Task1. 最小 MVP — 实施计划

> 目标：完成一个可运行的前端最小原型，让用户在浏览器中打开摄像头后，系统能够识别人体姿态，针对“开合跳”进行实时计数，并显示运动时长、频率、动态 MET、估算热量和基础置信度。
> 本任务只验证核心视觉链路，不做完整后端、用户系统、历史记录、复杂模型训练和多动作覆盖。

---

## 一、背景和目标

本项目定位为：

```text
基于视觉动作捕捉的运动强度评估与热量估算系统
```

项目长期目标是通过摄像头识别用户动作，结合动作频率、幅度、用户体重和 MET 模型，输出运动强度、估算热量和动作反馈。

但当前是全新项目，没有代码基础。第一阶段最重要的不是把系统做完整，而是验证最关键的技术风险：

```text
1. 浏览器能否稳定打开摄像头。
2. MediaPipe 能否稳定识别人并输出关键点。
3. 前端能否实时绘制人体骨架。
4. 能否用规则算法判断开合跳状态。
5. 能否根据状态变化稳定计数。
6. 能否根据体重、时长和 MET 公式估算热量。
```

因此 Task1 只做最小 MVP：

```text
摄像头预览
  ↓
人体姿态识别
  ↓
骨架绘制
  ↓
开合跳计数
  ↓
频率与热量估算
  ↓
基础置信度展示
```

## 二、任务范围

### 本次要做

```text
1. 初始化一个前端项目。
2. 接入浏览器摄像头。
3. 接入 MediaPipe Pose Landmarker。
4. 在 Canvas 上绘制视频画面、人体关键点和骨架连线。
5. 实现开合跳的打开/闭合状态判断。
6. 实现“闭合 → 打开 → 闭合 = 1 次”的计数逻辑。
7. 统计运动时长、次数、每分钟频率。
8. 提供体重输入。
9. 根据动态 MET 估算热量。
10. 展示基础置信度和主要影响因素。
11. 提供清晰的启动、识别中、未识别、摄像头不可用等状态。
```

### 本次不做

```text
1. 不做后端服务。
2. 不做数据库。
3. 不做登录和用户系统。
4. 不保存历史运动记录。
5. 不训练 LSTM、Transformer、ST-GCN 等模型。
6. 不覆盖高抬腿、深蹲等其他动作。
7. 不做医学级精确热量计算。
8. 不做多人同时入镜识别。
9. 不做移动端 App。
```

```text

5. 实时指标面板：
   - 开合跳次数
   - 运动时长
   - 当前频率
   - 动态 MET
   - 估算 kcal
```

## 五、核心算法设计

### 1. 开合跳状态判断

开合跳可以先按两个信号判断：

```text
手臂是否打开
双脚是否打开
```

可用关键点：

```text
肩膀
手腕
髋部
脚踝
```

初始规则建议：

```text
1. 双脚间距大于肩宽的一定倍数，认为脚打开。
2. 双手腕高度接近或高于肩膀，且手腕横向距离较大，认为手打开。
3. 手脚同时打开，状态为 open。
4. 手脚回到较窄位置，状态为 closed。
```

为了减少抖动，建议增加：

```text
1. 连续若干帧满足条件才切换状态。
2. open 和 closed 使用不同阈值，形成滞后区间。
3. 关键点置信度过低时不更新动作状态。
```

### 2. 计数规则

计数状态机：

```text
初始状态：waiting_closed

检测到 closed：
  进入 ready

ready 状态检测到 open：
  进入 opened

opened 状态检测到 closed：
  count + 1
  回到 ready
```

这样可以避免用户一开始站在打开状态时误计数。

### 3. 频率计算

建议按最近一段时间或整体运动时长计算：

```text
frequency = count / elapsedMinutes
```

展示时可以保留 1 位小数：

```text
18.5 次/分钟
```

运动时长从用户点击“开始”后开始计时。暂停时停止计时，重置时清空数据。

### 4. 动态 MET

Task1 只需要简单规则：

```text
开合跳低频：6.0 MET
开合跳中频：8.0 MET
开合跳高频：10.0 MET
```

频率分段可先设为：

```text
0 - 30 次/分钟：低频
30 - 50 次/分钟：中频
50 次/分钟以上：高频
```

后续可以根据真实测试调整。

### 5. 热量估算

使用标准 MET 公式：

```text
总消耗 kcal = MET × 3.5 × 体重kg ÷ 200 × 运动分钟数
```

如果用户没有输入体重，默认值可以设为：

```text
60 kg
```

页面上要允许用户修改体重。

### 6. 基础置信度

Task1 的置信度只需要简单可解释：

```text
高：全身关键点稳定，动作周期连续，人体基本居中。
中：偶尔丢点或画面角度偏，但仍能计数。
低：关键点缺失明显、身体未全身入镜、动作状态不稳定。
```

影响因素建议显示：

```text
1. 关键点完整率。
2. 是否全身入镜。
3. 动作周期是否稳定。
4. 摄像头角度是否明显偏侧。
```

## 七、验收标准

### 功能验收

```text
1. npm install 后可以安装依赖。
2. npm run dev 后可以打开前端页面。
3. 页面能请求摄像头权限。
4. 摄像头授权后能显示实时画面。
5. 姿态模型加载后能在人体上绘制关键点和骨架。
6. 没有人入镜时有明确提示。
7. 用户做开合跳时，系统能实时增加次数。
8. 开始、暂停、重置按钮行为正确。
9. 修改体重后，热量估算会随之变化。
10. 页面能显示次数、运动时长、频率、MET、估算 kcal 和置信度。
```

### 算法验收

```text
1. 用户站立不动时，不应持续误计数。
2. 用户完成一次明显开合跳时，应增加 1 次。
3. 手脚动作不完整时，可以不计数。
4. 关键点丢失时，不应继续错误累加。
5. 频率变化后，MET 档位能随之变化。
```

### 工程验收

```text
1. TypeScript 编译通过。
2. 前端构建通过。
3. 摄像头和 MediaPipe 资源加载失败时有错误提示。
4. 组件卸载时停止摄像头流和动画循环。
5. 不在控制台持续刷无意义错误。
```

建议检查命令：

```powershell
npm.cmd --prefix frontend run build
```

## 八、边界和默认假设

```text
1. Task1 只支持单人入镜。
2. Task1 只支持开合跳。
3. Task1 只做浏览器端 Demo。
4. Task1 不保存任何历史数据。
5. Task1 热量结果是估算值，不是医学级测量。
6. 摄像头需要能看到用户全身。
7. 用户应尽量正对摄像头。
8. 光线过暗、遮挡严重、衣物过宽松时，识别准确度可能明显下降。
9. 第一版阈值允许根据真实测试手动调整。
10. MediaPipe 模型资源可以先走 CDN，离线资源管理后续再做。
```
