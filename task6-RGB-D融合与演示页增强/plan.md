# Task6：RGB-D 融合与演示页增强

## 本 task 要完成的内容

Task6 承接 Task5 已跑通的 3D 摄像头能力，将 RGB-D depth 从 /#/pose 的调试信息推进到 /#/demo 的演示指标和置信度解释中。重点不是继续研究 3D 模型，而是让真实 depth 服务运动强度估算：前后方向动作证据、空间幅度、估算置信度和外场演示可读性。

当前主链路仍是 MediaPipe Pose / Hand Landmarker。OpenNI depth server 作为本地补充服务，根据 MediaPipe 的关键点 x/y 返回 depth/z 信息。

## 预期结果

1. /#/demo 能显示 depth 服务状态，明确当前是否接入 RGB-D。
2. /#/demo 能展示前后方向基础动作证据，例如手前移、手后移、接近身体平面。
3. depth 证据能参与估算置信度，但不直接抢占原有动作识别标签。
4. 侧身、躯干不可见、depth server 未启动、采样疑似背景时，界面能给出质量提示。
5. 保留 /#/pose 作为调试页，/#/demo 作为外场展示页。
6. 整理启动和验收流程，方便后续演示时同时启动 frontend 与 depth server。

## 当前结论

1. Task6 已将 RGB-D depth 接入 /#/demo，支持服务状态、前后方向动作证据、估算置信度原因和质量提示。
2. depth 证据不抢占当前基本动作或组合动作名称，只作为前后方向证据和估算质量因素。
3. 侧身、躯干不可见和疑似背景采样已能降级提示，但并未完全解决侧身识别问题。
4. 后续 Task7 应重点优化 /#/demo 的外场大屏视觉层级、用户体验和路人可读性。

## 可能的实现方案和技术栈

1. 前端：Vue 3 + TypeScript，优先复用 PoseView 中已有的 depth 请求、采样来源、z 轴差值和动作证据计算逻辑。
2. 后端：继续使用 task5-SAM 3D Body尝试/openni_depth_server.py，不在 Task6 重写服务。
3. 融合方式：MediaPipe 负责 2D 关键点和动作/运动量；depth server 补充前后方向证据和质量状态。
4. 置信度：depth 可用且与动作方向一致时提高前后动作证据；depth 缺失、疑似背景、躯干基准不稳时降低估算置信度。
5. 演示页：/#/demo 不展示过多调试字段，只展示路人可读的状态、动作、强度、空间幅度、估算消耗和质量提示。

## 边界

1. 不在 Task6 接入 SAM 3D Body 实时推理。
2. 不要求解决所有侧身和遮挡问题，只需要识别风险并反馈置信度。
3. 不把 depth 作为唯一动作识别依据，避免在 registration 未完全解决前过度依赖 z 值。
4. kcal 仍偏演示可解释，不追求商业级真实代谢估算。
5. registration 开启失败仍作为独立风险项记录，不阻塞 /#/demo 的 raw depth 演示融合。
