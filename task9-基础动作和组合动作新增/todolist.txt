# Task9 基础动作和组合动作新增 — 实施清单

> 状态：已开始。  
> 目标：扩展基础动作和组合动作库，逐个动作测试、调参和记录误报漏报。  
> 原则：基础动作可以并存，组合动作由基础动作 + 姿态关系 + 时序规则组成；不把动作识别做成单一互斥分类。  

---

## Phase 1: 动作库范围确认

- [x] 1.1 明确基础动作和组合动作边界
      → 基础动作: 可复用的短动作证据，例如手向前、双脚打开、下蹲
      → 组合动作: 由多个基础动作、姿态关系和时序规则组成，例如出拳、开合跳、深蹲
      → 要求: 基础动作不被组合动作覆盖，组合动作只引用基础动作证据
      → 已确认: 基础动作允许多项并存，不再把“当前动作”理解为单一互斥分类
      → 已确认: 组合动作必须包含时序或持续稳定性，不能只依赖单帧姿势

- [x] 1.2 确认第一批基础动作
      → 上肢: 左手上抬、右手上抬、双臂上抬、手臂回落、手向前移动、手向后移动、手水平摆动、手臂伸展、手臂弯曲
      → 下肢: 双脚打开、双脚合拢、左抬腿、右抬腿、膝盖上抬、下蹲、起身
      → 躯干: 身体前倾、身体后仰、身体左倾、身体右倾、身体左转、身体右转
      → 头部: 点头、摇头、左转头、右转头
      → 已确认: 第一批先覆盖上肢、下肢、躯干、头部四类；Phase 2 从现有 /#/pose 与 /#/demo 已有动作候选开始扩展
      → 已确认: “双脚合拢”和“下蹲”虽保留为基础动作，但进入 Phase 2 优先校准，避免频繁误显示

- [x] 1.3 确认第一批组合动作
      → 静息、挥手、出拳、开合跳、高抬腿、深蹲、原地摆臂
      → 已确认: 静息是低运动组合状态，用于演示页站立不动时展示
      → 已确认: 挥手、出拳、开合跳、高抬腿、深蹲作为第一批重点组合动作；原地摆臂作为补充动作

- [x] 1.4 记录 Task9 启动原因和首个校准问题
      → 已确认: “尽可能增加动作识别”需要动作库、阈值、状态机、置信度和逐动作验收，已超出 Task7 展示优化和 Task8 识别模型研究范围，因此单独设为 Task9
      → 已完成: Task9 从基础动作和组合动作新增开始推进
      → 已记录: 首个待校准问题是 /#/demo 的“当前基本动作”中“双脚合拢”和“下蹲”出现过于频繁，需要在下肢基础动作评分阶段优先处理

- [x] 1.5 Phase 1 收尾
      → 已完成: Phase 1 完成动作库范围确认
      → 已确认: 本阶段只更新文档和验收口径，不改前端识别代码，因此 /#/demo 画面不应出现新的动作识别行为变化
      → 下一步: 进入 Phase 2，对基础动作评分做代码实现和校准

## Phase 2: 基础动作评分扩展

- [x] 2.1 上肢基础动作评分
      → 使用肩、肘、腕、手部关键点和 depth 前后方向证据
      → 输出每个动作 0~100 独立分数和原因
      → 已完成: /#/demo 和 /#/pose 同步扩展左手上抬、右手上抬、双臂上抬、手臂回落、手向前移动、手向后移动、手水平摆动、手臂伸展、手臂弯曲
      → 已完成: 手向前 / 手向后优先使用 RGB-D depth 证据；未连接 depth 时保守不显示前后移动
      → 已完成: 手臂伸展不再只靠肘角，额外叠加上肢活动证据，减少静息垂手时的误报

- [x] 2.2 下肢基础动作评分

      → 重点修正“双脚合拢”置信度偏高问题

      → 已完成: /#/demo 和 /#/pose 同步校准下肢基础动作评分
      → 已完成: “双脚合拢”不再只靠脚踝距离小判断，还需要脚踝 / 膝盖距离收窄和下肢运动证据
      → 已完成: “下蹲”不再只靠髋部轻微下移判断，还需要膝髋结构变化或稳定下蹲姿态
      → 已完成: 基础动作候选展示阈值从 20 提高到 35，减少弱证据动作频繁上榜
      → 已验证: npm.cmd --prefix frontend run build 通过

- [x] 2.3 躯干和头部基础动作评分
      → 使用肩髋相对关系、头部少量点和时序变化

      → 已完成: /#/demo 和 /#/pose 同步接入鼻尖、左右耳少量头部点，用于点头、摇头、左转头、右转头
      → 已完成: 躯干使用肩髋中心相对关系和左右肩 z 差，输出身体前倾、后仰、左倾、右倾、左转、右转
      → 已完成: 躯干 / 头部评分第一版偏保守，主要用于低强度活动解释，后续根据实测误报再调阈值

- [x] 2.4 Phase 2 进展和收尾记录
      → 已完成: 2.1 上肢、2.2 下肢、2.3 躯干 / 头部基础动作评分扩展

## Phase 3: 组合动作状态机

- [x] 3.1 静息组合状态
      → 检测到人体 + 运动量低 + 持续稳定
      → 用于 /#/demo 站立不动时展示，不作为高 cal 动作
      → 已完成: /#/demo 增加静息状态机，低运动量需持续约 600ms 才进入“静息”
      → 已完成: 进入静息后，弱上肢候选不会立即打断；需明显活动证据持续约 280ms 才退出
      → 已完成: 解决“上肢活动”因轻微手部 / 关键点抖动过多抢占“静息”的问题
      → 已验证: npm.cmd --prefix frontend run build 通过

- [x] 3.2 挥手组合动作复核
      → 基于手水平摆动、周期和可见率
      → 记录与手向前 / 出拳的混淆情况
      → 已完成: /#/demo 增加挥手组合动作状态机，基于“手水平摆动”、腕点横向换向、上肢位移和腕点可见率判断
      → 已完成: 挥手优先于普通“上肢活动”展示，但不抢占“静息”、强“全身活动”和开合跳优先分支
      → 已记录: 若 RGB-D 前后移动和手臂伸展证据同时偏强，会视为疑似出拳/前后移动干扰；只有横向摆动和换向证据足够时才保留挥手
      → 实测问题: 单手挥手较难触发“挥手”，双手挥手更容易被识别；后续需要降低单侧横向换向门槛或单独做单手挥手证据
      → 已处理: 手水平摆动评分改为更看重单侧强横向位移，降低单手挥手被双手平均稀释的问题
      → 已验证: npm.cmd --prefix frontend run build 通过

- [x] 3.3 出拳组合动作
      → 手向前移动 + 手臂伸展 + 回收
      → 优先结合 RGB-D 前后方向证据
      → 已完成: /#/demo 增加出拳组合动作状态机，基于手向前移动、手臂伸展、上肢位移和回收证据判断
      → 已记录: 出拳第一版依赖 RGB-D 前后方向证据；未连接 depth 时会偏保守，可能更多显示为“上肢活动”
      → 实测问题: 出拳触发较难，且可能在挥手过程的某个瞬间误触发；后续需要引入“握拳 + 身体水平前侧 + 前向位移”的联合条件，并与斜上方挥手分开
      → 已处理: 出拳状态机加入握拳证据和“手腕位于身体水平前侧附近”约束，挥手斜上方动作不再轻易触发出拳
      → 已验证: npm.cmd --prefix frontend run build 通过

- [x] 3.4 开合跳组合动作
      → 双臂上抬 / 回落 + 双脚打开 / 合拢
      → 保留原开合跳状态机，并重新校准基础动作阈值
      → 已完成: 保留原开合跳优先分支，同时补充基础动作证据版开合跳判断
      → 已完成: 双臂上抬 + 双脚打开、手臂回落 + 双脚合拢都可作为开合跳组合证据
      → 已记录: 当前仍是展示状态机，不做完整计数；全身入镜、脚踝可见和上下肢同步性会显著影响识别
      → 已验证: npm.cmd --prefix frontend run build 通过

- [x] 3.5 高抬腿组合动作
      → 左 / 右膝盖上抬 + 左右交替周期
      → 记录是否受摄像头角度和下肢入镜影响
      → 已完成: /#/demo 增加高抬腿组合动作状态机，基于左抬腿、右抬腿、膝盖上抬、下肢位移和左右交替窗口判断
      → 已记录: 若下肢未完整入镜、侧身或膝盖关键点不稳，高抬腿可能退化为“下肢活动”
      → 已验证: npm.cmd --prefix frontend run build 通过

- [x] 3.6 深蹲组合动作
      → 下蹲 + 起身 + 髋膝高度变化
      → 记录侧身、遮挡和躯干参考不稳问题
      → 已完成: /#/demo 增加深蹲组合动作状态机，基于下蹲、起身和下肢位移判断
      → 已记录: 当前依赖髋、膝、踝关键点和正面姿态；侧身、遮挡、躯干参考不稳时需要后续逐动作校准
      → 已验证: npm.cmd --prefix frontend run build 通过

- [x] 3.7 静息兜底和通用组合展示修正
      → 已发现: /#/demo 在缺少 leftWrist 等组合动作关键点时，会提前 reset 为“未识别组合动作”，导致静止站立但手腕点缺失时没有显示“静息”
      → 已完成: 检测到人体且当前强度为静止时，即使组合动作关键点不足，也优先显示“静息”，detail 中保留关键点不足原因
      → 已发现: /#/demo 的组合动作展示仍受旧“开合跳所需关键点完整”分支影响，例如基础动作已识别为“双臂上抬”，但 leftHip 等开合跳关键点缺失时仍显示“未识别组合动作”
      → 已完成: 修正为通用组合动作展示逻辑；开合跳只是优先候选之一，上肢基础动作显示“上肢活动”，下肢基础动作显示“下肢活动”，上下肢同时存在显示“全身活动”，低运动量显示“静息”

- [x] 3.8 全身活动判标收紧
      → 已发现: 单纯动手时，轻微下肢候选或整体活动分数兜底会让 /#/demo 在“上肢活动”和“全身活动”之间跳动
      → 已完成: 只有上肢和下肢基础动作分数都达到较强证据，并且最近帧上肢 / 下肢位移都达标时才显示“全身活动”
      → 已完成: 整体活动分数兜底改为“活动中”，不再直接判为全身
      → 已验证: npm.cmd --prefix frontend run build 通过

- [x] 3.9 Phase 3 组合动作集中记录
      → 已完成: /#/demo 静息组合状态机，低运动量候选需持续约 600ms 才显示“静息”；明显活动证据需持续约 280ms 才退出静息
      → 已完成: /#/demo 挥手组合动作状态机，以“手水平摆动”、腕点横向换向次数、上肢位移和腕点可见率作为主要证据，并设置约 850ms 展示保持
      → 已完成: 挥手加入第一版混淆抑制；当“手向前 / 手向后”和“手臂伸展”同时较强时，优先认为存在出拳或前后移动干扰，只有横向换向或横摆证据足够强时才继续显示“挥手”
      → 已记录: 单手挥手较难触发“挥手”，双手挥手更容易识别；后续应单独增强单手挥手证据
      → 已记录: 出拳触发较难，且容易在挥手的某个瞬间被误识别为“出拳”；后续出拳应要求“手握拳 + 手出现在身体水平前侧 + 明确前向位移”
      → 已完成: 挥手基础评分改为更看重单侧手腕的强横向摆动；出拳组合动作加入握拳分数和手腕水平前侧分数
      → 已完成: 为 /#/demo 补齐出拳、开合跳、高抬腿、深蹲四个组合动作的第一版状态机；当前版本先用于集中验收，后续根据误报 / 漏报逐动作调阈值

      → 已完成: /#/demo 的“估算消耗”改为演示优先口径；只要检测到人体，就展示 cal 反馈，避免路人轻微挥手、抬手或小幅动作时长时间显示 0.00

      → 已确认: 数值会随当前动作刷新和回落，更适合路人看到“我刚刚动了一下，屏幕立刻有反馈”的演示场景
