# Task9：基础动作和组合动作新增

## 本 task 要完成的内容

Task9 负责在现有实时识别链路上扩展动作库，明确区分“基础动作”和“组合动作”，并对每个动作进行测试、阈值调整和误报记录。Task9 不以模型研究为主，也不负责 /#/demo 的视觉布局；它承接 Task7 的展示入口和 Task8 的识别能力研究结果，把系统从少数示例动作扩展为更完整的动作证据体系。

## 预期结果

1. 系统能输出更多基础动作候选，而不是只显示少数已有动作。
2. 每个基础动作都有可解释的输入特征，例如关节距离、角度、速度、前后 depth、时序稳定性。
3. 组合动作通过基础动作、姿态关系和时间顺序形成，不直接抢占基础动作信号。
4. /#/demo 能展示“当前基本动作”和“当前组合动作”，但算法调试和逐项阈值验证优先放在 /#/pose。
5. 每个新增动作都要记录测试结果、误报场景、漏报场景和是否进入演示页。

## 基础动作候选

1. 上肢：左手上抬、右手上抬、双臂上抬、手臂回落、手向前移动、手向后移动、手水平摆动、手臂伸展、手臂弯曲。
2. 下肢：双脚打开、双脚合拢、左抬腿、右抬腿、膝盖上抬、下蹲、起身。
3. 躯干：身体前倾、身体后仰、身体左倾、身体右倾、身体左转、身体右转。
4. 头部 / 面部：点头、摇头、左转头、右转头。

## Phase 1 范围确认结果

1. 基础动作不是唯一分类结果，可以同时存在多个独立分数。例如“手向前移动”和“手臂伸展”可以同时成立，它们后续可共同组成“出拳”。
2. 组合动作不是单帧姿势标签，而是由基础动作、姿态关系、时间顺序和持续稳定性组成。例如“深蹲”不能只靠一帧“下蹲”，还需要下蹲到起身的周期。
3. “静息”作为组合状态保留，用于检测到人体但运动量持续较低时展示；它不作为高 kcal 动作，也不应覆盖后续真实运动。
4. 第一批动作优先服务 /#/demo 外场展示和 /#/pose 调试，不追求一次覆盖所有姿势。
5. Task9 的第一个算法优化重点是校准下肢基础动作，尤其是“双脚合拢”和“下蹲”在正常站立时出现过于频繁的问题。

## 组合动作候选

1. 静息：检测到人体 + 运动量低 + 持续稳定，用于演示页低运动状态展示。
2. 挥手：手水平摆动 + 持续周期 + 手部可见。
3. 出拳：手向前移动 + 手臂伸展 + 回收，可结合 RGB-D 前后方向证据。
4. 开合跳：双脚打开 / 合拢 + 双臂上抬 / 回落 + 周期状态机。
5. 高抬腿：左 / 右膝盖上抬 + 躯干稳定 + 左右交替周期。
6. 深蹲：下蹲 + 起身 + 膝髋高度变化 + 躯干参考稳定。
7. 原地摆臂：左右手或双臂持续摆动 + 中低强度活动累计。

## 优先级

1. P0：先扩展基础动作库，保证多动作证据能同时存在。
2. P1：优先完善静息、挥手、出拳、开合跳、高抬腿、深蹲六个组合动作。
3. P2：补充头部和躯干动作，主要用于演示丰富度、质量反馈和低强度活动解释。
4. P3：结合 Task8 的人体分割、RGB-D 区域采样或人体建模结果，对动作阈值做个体化校准。

## 可能的实现方案和技术栈

1. 前端：Vue 3 + TypeScript，继续复用 /#/pose 和 /#/demo 的 MediaPipe Pose / Hand / Face 结果。
2. 识别：规则算法 + 多证据评分 + 状态机；不把动作识别改成单一互斥分类。
3. RGB-D：继续使用 OpenNI depth server，为手向前 / 后、身体前倾 / 后仰等动作提供 z 轴证据。
4. 调试：/#/pose 展示每个基础动作的分数、阈值和原因；/#/demo 只展示路人可读结果。
5. 测试：每个动作至少记录正例、反例、容易混淆动作和建议阈值。

## 边界

1. Task9 不负责研究 DensePose、SAM 3D Body、人体 mesh 等模型路线，这些属于 Task8 或后续人体建模任务。
2. Task9 不追求一次覆盖所有动作，优先保证少量动作稳定、可解释、可验收。
3. Task9 不直接追求 kcal 精准，kcal 仍按演示可解释口径，重点是动作时间、频率、幅度、强度和置信度。
