# 基于视觉动作捕捉的运动强度评估与热量估算系统

## 1. 项目定位

本项目建议定位为：

> 基于视觉动作捕捉的运动强度评估与热量估算系统

不建议直接定位为“精准卡路里计算系统”。纯视觉方案可以较好识别动作幅度、频率、节奏、完成次数和动作标准度，但对真实代谢消耗只能做估算。因此，系统核心价值应从“精确计算热量”调整为：

1. 不依赖手环、手表、心率带等穿戴设备；
2. 通过摄像头自动识别动作类型和完成次数；
3. 根据动作频率、幅度和用户体重动态估算运动消耗；
4. 给出动作质量反馈，例如膝盖角度、下蹲深度、左右不平衡；
5. 输出热量估算置信度，并说明影响估算质量的因素。

更准确的产品表达是：动作识别、次数统计、强度评估、热量估算和动作纠错的综合系统，而不只是一个热量计算器。

## 2. 第一版目标与 MVP 范围

第一版不建议覆盖所有健身动作，应先选择视觉特征明显、规则可解释、演示效果稳定的动作。建议 MVP 只做 3 个动作：

| 动作 | 适合原因 | 可计算指标 |
| --- | --- | --- |
| 开合跳 | 位移明显，节奏明显，适合视觉识别 | 次数、频率、节奏、动态 MET |
| 高抬腿 | 强度变化明显，速度与热量相关性较好 | 抬腿高度、左右频率、持续时间 |
| 深蹲 | 能展示动作标准度反馈价值 | 膝关节角度、髋膝联动、下蹲深度、次数 |

第一版主链路：

```text
摄像头识别人
  ↓
提取人体关键点
  ↓
判断动作类型
  ↓
统计次数、频率、幅度
  ↓
映射动态 MET
  ↓
输出热量估算、置信度和动作反馈
```

暂不建议第一版直接做俯卧撑、平板支撑、负重训练、瑜伽和拳击。原因是它们可能存在遮挡多、静态发力明显、负重不可见或动作复杂度过高等问题，容易让纯视觉系统暴露误差。

## 3. 核心技术栈

### 3.1 姿态估计

姿态估计是系统的视觉基础，用于从视频流中实时提取人体骨骼关键点坐标。

推荐方案：

| 技术 | 特点 | 适用性 |
| --- | --- | --- |
| MediaPipe Pose Landmarker | 支持 Web 和移动端，能输出 2D 图像坐标和 3D world coordinates | 第一版首选 |
| YOLO-Pose | 目标检测体系成熟，适合多人或复杂场景扩展 | 可作为后续备选 |
| OpenPose | 精度较高，但计算开销较大 | 不建议第一版使用 |

第一版建议优先使用 MediaPipe。它更适合端侧实时推理，部署复杂度也较低。

### 3.2 动作识别与分类

动作识别用于根据关键点序列判断用户正在做什么动作。

第一版建议采用规则算法，而不是直接训练复杂模型。规则算法更容易快速跑通系统，也便于解释和调试。后续可以再升级到 LSTM、Transformer 或 ST-GCN。

### 3.3 运动学量化

运动学量化用于把关键点轨迹转化为可计算指标，包括：

1. 动作次数；
2. 动作周期；
3. 单位时间频率；
4. 关键关节角度；
5. 动作幅度；
6. 节奏稳定性；
7. 左右侧差异。

初期不建议复杂计算加速度和完整物理做功。更稳妥的方式是使用动作频率法、关键点相对距离和关节角度变化来评估运动强度。

## 4. 系统架构设计

本项目适合采用：

> 端侧推理 + 后端记录 + 可视化面板

推荐架构：

```text
摄像头视频流
  ↓
前端 Pose Estimation
  ↓
关键点坐标序列
  ↓
动作识别 / 次数统计 / 频率计算
  ↓
动态 MET 映射
  ↓
卡路里估算
  ↓
本地实时展示 + 后端存储
```

后端不建议承担实时视频推理，否则会增加延迟、带宽压力和部署复杂度。后端主要负责：

1. 用户信息管理；
2. 运动记录存储；
3. 历史统计；
4. MET 对照表维护；
5. 模型和规则参数配置；
6. 管理端数据看板。

推荐技术栈：

| 模块 | 推荐技术 |
| --- | --- |
| 前端 | Vue 或 React + MediaPipe JS |
| 后端 | FastAPI |
| 数据库 | SQLite、MySQL 或 PostgreSQL |
| 算法 | MediaPipe + 规则算法，后续扩展 LSTM、Transformer、ST-GCN |
| 部署 | 浏览器端优先，后续可扩展到移动端 |

## 5. 热量估算模型

### 5.1 标准 MET 公式

系统可以继续使用 MET 体系，但后端实际计算建议采用更标准的公式：

```text
kcal/min = MET × 3.5 × 体重kg ÷ 200
```

即：

```text
总消耗 kcal = MET × 3.5 × 体重kg ÷ 200 × 运动分钟数
```

该公式天然包含体重因素。体重越高，在同样 MET 和运动时间下，估算消耗越高。

### 5.2 动态 MET 映射

一个动作不应只对应固定 MET 值，而应根据动作强度动态调整。建议采用三层结构：

```text
动作类型 → 基准 MET
动作频率 → 强度修正
动作幅度 → 完成质量修正
```

简化版本可以先实现为：

```text
动态 MET = 动作基础 MET + 频率修正值
```

后续再扩展为：

```text
动态 MET = 基准 MET × 频率系数 × 幅度系数 × 动作完成度系数
```

示例：

| 动作 | 低强度 | 中强度 | 高强度 |
| --- | --- | --- | --- |
| 深蹲 | 4.0 MET | 5.5 MET | 7.0 MET |
| 开合跳 | 6.0 MET | 8.0 MET | 10.0 MET |
| 高抬腿 | 5.0 MET | 7.0 MET | 9.0 MET |

实际开发前需要查阅运动医学文献，建立动作与低、中、高强度 MET 的基准对照表。2024 Adult Compendium 可作为主要参考来源。

## 6. 第一版动作识别规则

### 6.1 开合跳

判断逻辑：

1. 双手从身体两侧上举到头顶附近；
2. 双脚间距从窄变宽；
3. 手脚开合呈周期性同步变化；
4. 完成一次“闭合 → 打开 → 闭合”，计为 1 次。

可反馈问题：

1. 手臂没有完全打开；
2. 双脚开合幅度不足；
3. 节奏不稳定；
4. 动作中断。

### 6.2 高抬腿

判断逻辑：

1. 左右膝盖交替上升；
2. 膝盖高度超过髋部附近的比例阈值；
3. 单位时间内统计左右抬腿次数；
4. 根据频率分为低、中、高强度。

可反馈问题：

1. 抬腿高度不足；
2. 左右频率不一致；
3. 身体晃动过大；
4. 动作节奏不稳定。

### 6.3 深蹲

判断逻辑：

1. 髋部高度下降；
2. 膝关节角度变小；
3. 躯干角度变化处于合理范围；
4. 下降到一定深度后再回升，计为 1 次。

可反馈问题：

1. 下蹲深度不足；
2. 膝盖内扣；
3. 身体前倾过大；
4. 起身速度不稳定；
5. 左右受力不均。

## 7. 置信度机制

每次输出热量时，不应只显示单一数字，而应同时输出估算置信度和影响因素。

示例：

```text
本次估算消耗：43 kcal
置信度：中等
影响因素：摄像头角度偏侧、下半身关键点偶有丢失
```

置信度可以由以下因素计算：

1. 关键点完整率；
2. 人体是否全身入镜；
3. 是否多人入镜；
4. 光照是否稳定；
5. 动作识别概率；
6. 动作周期是否稳定；
7. 用户是否正对摄像头。

置信度机制可以把“系统存在误差”转化为“系统知道自己什么时候更可靠”，这对展示和产品可信度都很重要。

## 8. 用户档案与数据记录

系统至少需要用户录入体重，因为体重是 MET 公式的必要变量。

建议用户档案字段：

| 字段 | 用途 |
| --- | --- |
| 体重 | 热量估算必要变量 |
| 身高 | 后续可用于身体比例校准 |
| 年龄 | 后续可用于个体化估算 |
| 性别 | 后续可用于个体化估算 |
| 常用运动水平 | 后续可用于强度映射修正 |

运动记录建议保存：

1. 动作类型；
2. 运动时长；
3. 完成次数；
4. 平均频率；
5. 动态 MET；
6. 估算消耗；
7. 置信度；
8. 动作质量评分；
9. 主要反馈问题；
10. 时间戳。

## 9. 工程落地挑战

### 9.1 2D 摄像头的深度错觉

如果用户正对镜头做前后方向运动，例如直拳向前，画面中的 X 轴和 Y 轴位移可能很小。若不借助 3D 姿态模型，系统容易误判为低强度运动。

### 9.2 环境干扰

过暗光线、逆光、宽松衣物、遮挡和摄像头角度偏差，都会降低关键点定位质量，进而影响动作识别和热量估算。

### 9.3 端侧算力与功耗

手机或浏览器长时间运行实时姿态估计，可能带来发热和耗电问题。需要在帧率、模型复杂度和识别稳定性之间做平衡。

### 9.4 纯视觉方案的物理盲区

视觉系统无法直接感知真实阻力和肌肉发力。典型问题包括：

1. 空手深蹲和负重深蹲在视觉轨迹上可能相似，但真实消耗不同；
2. 平板支撑、靠墙静蹲等静态发力动作，画面位移很小但肌肉消耗明显；
3. 同体重用户也可能因肌肉量、体脂率和代谢效率不同而有真实消耗差异。

## 10. 系统适用边界与误差说明

文档中应明确说明：

1. 本系统估算的是运动热量，不是医学级能耗测量；
2. 本系统不替代专业运动医学设备或医疗判断；
3. 本系统不适合负重训练的精确热量评估；
4. 本系统不适合平板支撑、靠墙静蹲等静态发力动作的精确评估；
5. 本系统不适合多人同时入镜；
6. 本系统不适合严重遮挡、逆光、宽松衣物等场景；
7. 输出结果应以“估算消耗”和“置信度”形式呈现。

## 11. 后续优化方向

### 11.1 算法优化

从规则识别逐步升级到 LSTM、Transformer 或 ST-GCN，提高复杂动作识别能力和泛化能力。

### 11.2 多模态融合

接入手机 IMU、智能手表心率或蓝牙心率带。视觉负责动作识别，心率负责生理强度，两者结合会比纯视觉更可靠。

### 11.3 个体化校准

用户连续使用一段时间后，系统可以根据历史表现调整强度映射。例如同样 30 次/分钟的深蹲，新手和训练者的运动负担可能不同。

### 11.4 产品化功能

可以扩展训练计划、排行榜、历史趋势、动作纠错、AI 教练语音提示和课程模式。

## 12. 可选项目名称

1. 基于视觉动作捕捉的运动强度评估与热量估算系统；
2. AI 视觉运动教练与卡路里估算平台；
3. 基于人体姿态估计的无穿戴运动消耗评估系统；
4. VisionFit：基于视觉姿态识别的智能健身评估系统；
5. PoseBurn：视觉动作识别与动态热量估算系统。

推荐使用第 1 个名称，正式、稳妥，技术表达也更准确。
