# Task1. 最小 MVP — 实施计划

> 目标：完成一个可运行的浏览器端前端 MVP。用户打开页面后授权摄像头，系统实时识别人体姿态，针对“开合跳”完成计数，并显示运动时长、频率、动态 MET、估算热量和基础置信度。
> 本任务只验证视觉识别与热量估算的核心链路，不建设完整业务系统。

---

## 一、背景和目标

项目定位：

```text
基于视觉动作捕捉的运动强度评估与热量估算系统
```

Task1 是全新项目的第一个开发任务，核心目标是验证最关键的可行性：

```text
摄像头预览
  ↓
人体姿态识别
  ↓
骨架绘制
  ↓
开合跳计数
  ↓
频率与热量估算
  ↓
基础置信度展示
```

本阶段交付结果必须是一个可本地运行、可实际对着摄像头测试的 MVP 页面。

---

## 二、确定范围

### 本次实现

```text
1. 初始化 frontend 前端项目。
2. 使用浏览器摄像头作为输入源。
3. 使用 MediaPipe Pose Landmarker 获取人体关键点。
4. 使用 Canvas 绘制摄像头画面、关键点和骨架。
5. 只识别开合跳一个动作。
6. 实现开合跳 open / closed 状态判断。
7. 实现“closed → open → closed = 1 次”的计数规则。
8. 统计运动时长、开合跳次数、每分钟频率。
9. 支持用户输入体重。
10. 使用 MET 公式计算估算热量。
11. 展示基础置信度和影响因素。
12. 处理摄像头授权、模型加载、未检测到人体、错误等状态。
```

### 本次不实现

```text
1. 不做后端。
2. 不做数据库。
3. 不做登录、用户档案和历史记录。
4. 不训练任何动作识别模型。
5. 不支持高抬腿、深蹲或其他动作。
6. 不支持多人同时入镜。
7. 不做医学级热量测量。
8. 不做移动端 App。
9. 不做离线模型资源管理。
```

本任务固定使用：


项目落点：

```text

  src/
```


### MediaPipe 版本边界

Task1 必须固定 MediaPipe 相关版本：

```text
1. package.json 中 @mediapipe/tasks-vision 使用固定版本，不使用 ^ 或 ~ 浮动范围。
2. Pose Landmarker 模型 URL 使用明确版本路径。
3. README 或任务文档记录当前验证通过的 tasks-vision 版本、模型 URL 和浏览器环境。
```

这样可以避免后续 CDN 或包版本变化导致初始化 API、wasm 路径或关键点输出行为发生变化。

## 四、页面和交互

Task1 只做一个页面。页面打开后直接进入识别界面，不做首页、介绍页或营销页。

页面必须包含：

```text
1. 摄像头画面区域。
2. 骨架绘制层。
3. 开始、暂停、重置控制。
4. 体重输入。
5. 实时指标：
   - 次数
   - 运动时长
   - 次/分钟
   - 当前 MET
   - 估算 kcal
   - 置信度
6. 运行状态：
   - 等待摄像头授权
   - 正在加载模型
   - 已就绪
   - 识别中
   - 未检测到完整人体
   - 摄像头或模型错误
```

开始后才累计运动时长和次数；暂停后停止累计；重置后清空计数、时长、热量和状态机。

## 五、算法规则

### 坐标系统与镜像处理

Task1 固定区分两套坐标：

```text
算法坐标系：使用 MediaPipe 原始关键点坐标。
显示坐标系：允许镜像，让用户看到类似照镜子的画面。
```

实现规则：

```text
1. 视频可镜像显示，但动作判断必须使用统一的算法坐标系。
2. Canvas 绘制层必须与用户看到的视频方向保持一致。
3. 左右关键点语义以 MediaPipe 的人体解剖学 left/right 为准。
4. CSS transform: scaleX(-1) 只影响显示，不得导致关键点绘制和算法判断错位。
```

后续判断左右手、左右脚、左右不平衡时，必须继续遵守这套坐标约定。

### 推理频率控制

Task1 不允许每个渲染帧都无脑发起姿态推理。

固定规则：

```text
1. Canvas 画面循环使用 requestAnimationFrame 驱动。
2. MediaPipe 姿态推理目标频率为 15-30 FPS。
3. 上一次姿态推理未完成时，不发起下一次推理。
4. 页面记录并显示实际推理 FPS。
```

这里的 FPS 指 MediaPipe 实际推理频率，不等同于 Canvas 渲染刷新率。

### 关键点平滑处理

Task1 对关键点坐标做轻量平滑：

```text
1. 使用最近 3-5 帧移动平均。
2. 重点降低手腕、脚踝等关键点抖动。
3. 平滑后的关键点只用于动作判断。
4. 原始关键点仍用于检测状态、关键点完整率和调试记录。
```

平滑不能掩盖真实丢点。关键点缺失时，检测状态仍应按原始结果记录。

### 姿态识别调试面板

Task1 页面保留调试面板区域，用于显示识别链路内部状态。

Phase 3 必须显示：

```text
1. 是否检测到人体。
2. 关键点完整率。
3. 实际推理 FPS。
4. 模型加载状态。
5. 原始关键点和平滑关键点状态。
```

以下字段在后续阶段接入，不在 Phase 3 伪造：

```text
Phase 4 后显示：
- 当前动作状态: open / closed / unknown
- 脚部状态: open / closed
- 手臂状态: open / closed

Phase 5 后显示：
- 状态机状态: waiting_closed / ready / opened
- 最近一次计数时间
```

### 开合跳状态

状态只分为：

```text
unknown
closed
open
```

使用关键点：

```text
肩膀
手腕
髋部
脚踝
```

判定规则：

```text
1. 双脚间距大于肩宽阈值时，脚部为打开。
2. 双手腕高于或接近肩部，且横向距离达到阈值时，手臂为打开。
3. 手臂和脚部同时打开，整体状态为 open。
4. 双脚回到窄间距，手腕回落后，整体状态为 closed。
5. 关键点置信度不足时，不更新动作状态。
```

状态切换必须做防抖：

```text
1. 连续若干帧满足条件后才切换状态。
2. open / closed 使用不同阈值形成滞后区间。
```

### 计数状态机

计数状态固定为：

```text
waiting_closed
ready
opened
```

规则：

```text
1. 初始为 waiting_closed。
2. 检测到 closed 后进入 ready。
3. ready 检测到 open 后进入 opened。
4. opened 检测到 closed 后 count + 1，并回到 ready。
```

### 频率

频率按整体运动时长计算：

```text
frequencyPerMin = count / elapsedMinutes
```

页面展示保留 1 位小数。

### 动态 MET

Task1 固定使用开合跳三档 MET：

```text
0 - 30 次/分钟：6.0 MET
30 - 50 次/分钟：8.0 MET
50 次/分钟以上：10.0 MET
```

### 热量估算

固定公式：

```text
总消耗 kcal = MET × 3.5 × 体重kg ÷ 200 × 运动分钟数
```

体重默认值：

```text
60 kg
```

用户修改体重后，热量结果实时更新。

### 置信度

Task1 置信度只分为：

```text
高
中
低
```

判断依据：

```text
1. 关键点完整率。
2. 用户是否基本全身入镜。
3. 动作状态是否连续稳定。
4. 人体是否基本居中。
```

页面必须显示置信度等级和主要影响因素。

## 六、前端状态

运行状态固定为：

```text
idle       未开始
loading    摄像头或模型加载中
ready      摄像头和模型已就绪
running    正在识别和计数
paused     已暂停
error      出错
```

核心指标：

```text
weightKg
count
elapsedMs
frequencyPerMin
currentMet
calories
confidenceLevel
confidenceReasons
poseDetected
jumpingJackState
counterState
```

组件卸载或页面刷新前必须停止摄像头流和动画循环。

## 七、验收标准

### 功能验收

```text
1. npm install 能安装依赖。
2. npm run dev 能启动页面。
3. 页面能请求摄像头权限。
4. 摄像头授权后显示实时画面。
5. 模型加载后能绘制人体关键点和骨架。
6. 无人入镜时显示明确提示。
7. 用户做开合跳时次数能增加。
8. 站立不动时不会持续误计数。
9. 开始、暂停、重置行为正确。
10. 修改体重后估算 kcal 实时变化。
11. 页面显示次数、时长、频率、MET、kcal、置信度。
```

### 工程验收

```text
1. TypeScript 编译通过。
2. 前端构建通过。
3. 摄像头或模型加载失败时有错误提示。
4. 停止使用页面后摄像头流被释放。
5. 控制台没有持续刷屏错误。
```

检查命令：

```powershell
npm.cmd --prefix frontend run build
```

## 八、边界

```text
1. 本任务只交付浏览器端 Demo。
2. 本任务只支持单人、全身、正面入镜。
3. 本任务只支持开合跳。
4. 本任务不保存数据。
5. 本任务输出的是估算热量，不是医学级测量结果。
6. 光线过暗、遮挡、宽松衣物、摄像头角度偏侧会降低识别质量。
7. 阈值可在 Task1 开发中根据实测调整，但动作范围不得扩大。
8. MediaPipe 模型资源本地化放到后续任务。
```
