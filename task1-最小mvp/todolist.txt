# Task1 最小 MVP — 实施清单

> 目标：完成一个可运行的浏览器端前端 MVP。用户授权摄像头后，系统实时识别人体姿态，针对“开合跳”完成计数，并显示运动时长、频率、动态 MET、估算热量和基础置信度。  
> 原则：只验证核心视觉链路；不做后端、不做数据库、不保存历史记录、不扩展其他动作。
> 关键判断：Task1 成功标准不是“热量绝对精准”，而是“摄像头 → 姿态关键点 → 开合跳计数 → MET 热量估算”链路能真实跑通。

## Phase 4: 开合跳状态识别

- [ ] 4.1 提取开合跳所需关键点
      → 肩膀
      → 手腕
      → 髋部
      → 脚踝
      → 关键点缺失时返回 unknown
- [ ] 4.2 实现 open / closed 状态判断
      → 双脚间距大于肩宽阈值时，脚部为打开
      → 双手腕高于或接近肩部，且横向距离达到阈值时，手臂为打开
      → 手臂和脚部同时打开时，状态为 open
      → 双脚回到窄间距且手腕回落时，状态为 closed
- [ ] 4.3 增加状态防抖
      → 连续若干帧满足条件后才切换状态
      → open / closed 使用不同阈值形成滞后区间
      → 抖动帧不触发计数
- [ ] 4.4 验证站立状态
      → 用户站立不动时不持续误计数
      → 用户半动作或手脚不完整时可以不计数
      → 关键点丢失时不继续累加

## Phase 5: 开合跳计数和运动计时

- [ ] 5.1 实现计数状态机
      → 初始状态: waiting_closed
      → 检测到 closed 后进入 ready
      → ready 检测到 open 后进入 opened
      → opened 检测到 closed 后 count + 1，并回到 ready
- [ ] 5.2 实现开始、暂停、重置
      → 开始后才累计时间和次数
      → 暂停后停止累计
      → 重置后清空 count / elapsedMs / calories / 状态机
- [ ] 5.3 计算运动时长
      → 指标: elapsedMs
      → 暂停时间不计入运动时长
      → 页面展示为易读时间格式
- [ ] 5.4 计算每分钟频率
      → 公式: frequencyPerMin = count / elapsedMinutes
      → 页面展示保留 1 位小数
      → elapsedMinutes 为 0 时避免除零错误

## Phase 6: MET 和热量估算

- [ ] 6.1 实现体重输入
      → 默认值: 60 kg
      → 用户可修改
      → 非法输入需要回退或提示
- [ ] 6.2 实现开合跳三档 MET
      → 0 - 30 次/分钟: 6.0 MET
      → 30 - 50 次/分钟: 8.0 MET
      → 50 次/分钟以上: 10.0 MET
      → 频率变化后 MET 实时变化
- [ ] 6.3 实现热量估算公式
      → 公式: 总消耗 kcal = MET × 3.5 × 体重kg ÷ 200 × 运动分钟数
      → 修改体重后 kcal 实时更新
      → 暂停状态下 kcal 不继续增长
- [ ] 6.4 展示实时指标
      → 次数
      → 运动时长
      → 次/分钟
      → 当前 MET
      → 估算 kcal

## 待商讨记录区

### 开合跳阈值

Task1 开发中允许根据实测调整肩宽倍数、手腕高度阈值、脚踝距离阈值和防抖帧数，但动作范围不得扩大。

### CDN 模型资源

Task1 固定使用官方 CDN 加载 MediaPipe 模型。模型本地化、离线部署和资源版本锁定放到后续任务。

### 置信度粒度

Task1 只做高 / 中 / 低三档置信度。更细的评分、百分制和动作质量纠错放到后续任务。

### 后续动作

Task1 只支持开合跳。高抬腿、深蹲和动作质量评分必须另起任务处理。
