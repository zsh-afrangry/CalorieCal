# Task5 RGB-D 深度摄像头接入评估 — 实施清单

> 状态：已开始。  
> 目标：评估 Orbbec Astra Pro HD / ORBBEC Depth Sensor 是否能稳定输出真实 depth，并与当前 MediaPipe Pose / Hand 主链路融合。  
> 原则：RGB-D 优先，SAM 3D Body 暂作备选；先跑通 Python 最小 depth 读取，再考虑前端接入。  

---

      → MediaPipe Pose / Hand 仍作为实时前端主链路

## Phase 4: 结论和后续方案

- [x] 4.1 给出接入判断
      → 接入 /pose 调试 / 接入 /demo 演示 / 只保留探针 / 暂缓
      → 结论: RGB-D depth 不应只保留探针，建议继续接入 /pose 调试，并后续接入 /demo 演示
      → 理由: 正对摄像头时，depth 能补足 RGB 难以判断的前后方向运动，可服务“手前移/手后移/出拳/前倾后退”的证据评分

- [x] 4.2 如果暂缓，记录原因
      → 例如 depth 抖动大、registration 不稳定、Python 依赖复杂、实时性不足
      → 当前不整体暂缓 RGB-D；但 registration 开启失败、侧身躯干基准不稳定、Python 后端依赖复杂需要作为风险保留
      → 侧身问题后续可考虑更稳的 3D 骨架/人体 mesh、多相机、或使用更多躯干候选点和历史基准缓冲处理

- [x] 4.3 如果继续，拆分后续任务
      → Python depth 服务、前端关键点 depth 请求、/pose 调试展示、/#/demo 演示展示、置信度融合
      → 后续任务 1: 将 depth 前后证据纳入多运动证据评分，但先只作为独立证据，不抢占原有动作识别
      → 后续任务 2: /demo 增加 depth 状态提示，让商场屏幕能展示“前后运动/空间幅度/置信度”
      → 后续任务 3: 继续研究侧身和遮挡场景，判断是否需要 SAM 3D Body / 3D 骨架恢复作为补充

## 待商讨记录区

### Task5 的定位

Task5 是 RGB-D 深度接入评估，不是为了展示 3D 模型。它的价值必须落到运动时间、频率、幅度、强度、估算消耗和置信度，尤其是补足普通 RGB 摄像头难以判断的前后方向运动。

### SAM 3D Body 位置

SAM 3D Body 保留为备选研究方向。当前已有可用 Orbbec RGB-D 摄像头，因此优先评估真实 depth 硬件，而不是先从单张 RGB 估计人体 mesh。

### 侧身和遮挡记录

当前 RGB-D 前后证据在正对摄像头时基本可用，但侧身时容易丢失稳定躯干基准。后续 3D Body / 3D 骨架方案的价值主要在这里：改善侧身、遮挡、躯干不可见时的身体参考面估计，而不是替代已经可用的正面 depth 证据。
