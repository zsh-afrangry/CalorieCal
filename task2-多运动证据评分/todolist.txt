# Task2 多运动证据评分 — 实施清单

> 状态：进行中。  
> 目标：将 /#/gesture 从“单一当前动作”改成“多运动证据评分”，重点解决挥手被“握拳 / 手前移 / 手后移 / 出拳”等状态抢占，导致挥手计数中断或漏计的问题。  
> 原则：继续使用前端 MediaPipe Hand Landmarker + 规则算法；不引入后端、不训练模型、不追求商业级 kcal 精准估算。  
> 关键判断：Task2 成功标准不是“准确识别所有手势”，而是“挥手计数和热量估算不再被单一动作标签轻易打断，并且调试信息能解释为什么计数或不计数”。

## Phase 2: 多运动证据数据结构

- [x] 2.1 新增 motionEvidence 数据结构
      → 建议字段:
        - horizontalSwingScore: 横向摆动证据
        - depthMoveScore: 前后移动证据
        - fingerCurlScore: 手指弯曲 / 握拳证据
        - waveCycleScore: 挥手周期证据
        - stabilityScore: 关键点稳定性证据

      → 已完成: GestureView 新增 gestureCandidates 和候选状态展示

- [x] 2.3 新增 waveCounterSignal

## Phase 3: 多证据评分算法

- [x] 3.1 计算横向摆动证据
      → 输入: handMotionHistory 中 centerX 序列
      → 指标: xRange、xDirectionChanges、最近方向
      → 输出: horizontalSwingScore
      → 说明: 横向摆动是挥手计数的核心证据
      → 已完成: 基于 xRange 和 xDirectionChanges 计算

- [x] 3.2 计算前后移动证据
      → 输入: handMotionHistory 中 scale 序列
      → 指标: scaleChange、scaleRange
      → 输出: depthMoveScore
      → 说明: 手部变大 / 变小可以作为手向前 / 手向后的候选证据，但不应直接打断挥手计数
      → 已完成: 基于 scaleChange 计算

- [x] 3.3 计算手指弯曲 / 握拳证据
      → 输入: fingerCurl 序列
      → 指标: curlRange、当前 fingerCurl
      → 输出: fingerCurlScore
      → 说明: 可作为握拳、招手或手指变化的候选证据
      → 已完成: 基于 curlRange 计算

- [x] 3.4 计算挥手周期证据
      → 输入: xDirectionChanges、xRange、最近换向时间
      → 输出: waveCycleScore
      → 说明: 该分数用于 waveCounterSignal，不等同于 currentGesture
      → 已完成: 基于 horizontalSwingScore 和换向次数计算

- [x] 3.5 计算关键点稳定性证据
      → 输入: handCount、采样帧数量、最近帧连续性
      → 输出: stabilityScore
      → 说明: 用于降低残影、丢手、采样不足时的置信度
      → 已完成: 基于手部历史采样帧数计算

## Phase 4: 挥手计数逻辑改造

- [x] 4.1 修改 updateWaveCounter 入参

- [x] 4.2 允许短暂冲突状态不中断计数

- [x] 4.4 调整挥手置信度原因

## 待商讨记录区

### kcal 定位

当前项目距离正式商业化仍较远。Task2 的 kcal 继续偏“演示可解释”，用于说明运动时间、频率、幅度、强度和置信度如何影响估算结果；贴近真实消耗放到后续阶段。

### 多证据评分

“挥手 / 手向前 / 手向后 / 握拳”等状态不要求互斥。它们可以同时存在，并各自拥有独立置信度。Task2 的核心是避免单一标签抢占计数链路。
