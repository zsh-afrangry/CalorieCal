# Task12 双摄像头融合架构 — 实施清单

> 状态：进行中
> 目标：引入侧面摄像头，实现双视角融合的热量计算和动作识别，解决单摄像头遮挡问题。
> 原则：不做复杂 3D 重建，第一版采用可见度门槛 + 最大运动信号融合；保留调试开关；向后兼容单摄像头模式。
> 关键判断：Task12 成功标准是"用户侧身、扭腰时热量计算不中断"，不要求动作识别 100% 准确。

## Phase 2: 后端双视角融合引擎 ✅ 已完成（2026-06-13）

- [x] 2.0 realtime_engine.py 增加双视角独立状态
      → **关键问题**: 现有 _prev_landmarks、_prev_hip_height 等状态是单视角设计
      → **问题影响**: 如果 front/side 共享状态，side 的 motion_energy 会用 front 的 _prev_landmarks 计算，导致错误
      → **实现方案**: 分离关键状态（prev_landmarks 和 prev_hip_height）
        ```python
        self._prev_landmarks_front = None
        self._prev_landmarks_side = None
        self._prev_hip_height_front = None
        self._prev_hip_height_side = None
        self._prev_motion_active_view = None
        ```
      → 已完成: 新增 _prev_landmarks_front/_side、_prev_hip_height_front/_side、_prev_motion_active_view
      → 已完成: 单摄 push_frame 同步 front 状态，便于单/双摄切换

- [x] 2.1 realtime_engine.py 新增 push_dual_frame() 方法
      → 方法签名:
        ```python
        def push_dual_frame(
            self,
            landmarks_front: dict[str, dict] | None,
            landmarks_side: dict[str, dict] | None,
            timestamp_ms: float | None = None,
        ) -> dict:
        ```
      → 功能:
        - 分别对 front 和 side 调用 _extract_features()，**使用各自的 prev_landmarks/prev_hip_height**
        - 获取两路的 motion_energy、hip_height、core_visible_ratio
        - 用改进的融合策略（见 2.2）融合 motion_energy 和 hip_height
        - 用融合后的值计算 intensity → MET → kcal_increment
        - **动作识别只用 front**（见 2.3）
        - 返回 actions (front only)、features (merged)、calorie_summary、view_info
      → 已完成: 新增 push_dual_frame()，front 进入原有动作 buffer，side 只参与 motion_kcal 融合

- [x] 2.2 实现改进的融合策略（取较大运动信号）
      → **问题**: 原方案的可见度加权会低估运动（例如前后平移时正面投影压扁）
      → **实现策略**: 可见度门槛 + 取较大运动信号
        ```python
        VIS_THRESHOLD = 0.5
        if core_vis_front >= VIS_THRESHOLD and core_vis_side >= VIS_THRESHOLD:
            motion_e_merged = max(motion_e_front, motion_e_side)
            hip_h_merged = hip_h_front if motion_e_front >= motion_e_side else hip_h_side
            active_view = "front" if motion_e_front >= motion_e_side else "side"
        elif core_vis_front >= VIS_THRESHOLD:
            motion_e_merged = motion_e_front
            hip_h_merged = hip_h_front
            active_view = "front"
        elif core_vis_side >= VIS_THRESHOLD:
            motion_e_merged = motion_e_side
            hip_h_merged = hip_h_side
            active_view = "side"
        else:
            motion_e_merged = 0.0
            hip_h_merged = None
            active_view = "none"
        ```
      → 已完成: core_visible_ratio >= 0.5 的视角参与候选，双路可用时采用 motion_energy 较大视角
      → 已完成: active_view 切换时重置 hip delta 基准，避免 front/side 髋高跨视角相减

- [x] 2.3 动作识别和热量融合明确分离
      → **第一版明确边界**: 动作识别只用 front，side 只参与 motion_kcal
      → **原因**: 降低风险，热量计算立即受益，动作识别保持稳定
      → **实现**:
        ```python
        # 动作识别：只用 front
        if landmarks_front and core_vis_front >= 0.7:
            actions = self._run_front_actions(landmarks_front, features_front)
        else:
            actions = []

        # 热量计算：用融合后的 motion_e_merged
        if core_vis_merged >= 0.7:
            intensity = 3.0 * delta_hip_merged + 1.0 * motion_e_merged
            # ... MET → kcal_increment
        ```
      → 已完成: 抽出 _run_front_actions() 复用原有动作识别；push_dual_frame 只用 front 跑动作识别

- [x] 2.4 返回值新增 view_info 字段（调试用）
      → **重要**: view_info 是帧级字段，与 actions、calorie_summary 平级，不属于 actions 内部
      → 返回结构:
        ```json
        {
          "actions": [...],
          "calorie_summary": {...},
          "view_info": {
            "front_core_vis": 0.82,
            "side_core_vis": 0.35,
            "active_view": "front",
            "front_motion_e": 0.045,
            "side_motion_e": 0.032,
            "merged_motion_e": 0.045
          }
        }
        ```
      → 用途: 前端调试面板显示哪个视角在主导
      → 字段说明:
        - active_view: "front" / "side" / "none"（哪个视角的运动信号被采用）
        - front_motion_e / side_motion_e: 原始运动信号（调试用）
        - merged_motion_e: 融合后的运动信号（实际用于热量计算）
      → 已完成: push_dual_frame 返回 view_info，与 actions、calorie_summary 平级
      → 已验证: python3 -m py_compile backend/action_engine/realtime_engine.py backend/action_server.py 通过
      → 已验证: 引擎级 smoke test 中 side_motion_e 较大时 active_view=side

## Phase 5: 前端 WebSocket 消息发送

- [x] 5.1 useActionWs.ts 新增 sendDualFrame() 方法
      → 方法签名:
        ```typescript
        function sendDualFrame(
          frontNamed: Record<string, LandmarkPoint> | null,
          sideNamed: Record<string, LandmarkPoint> | null,
          timestampMs: number,
        ) {
          if (!ws || ws.readyState !== WebSocket.OPEN) return;
          ws.send(JSON.stringify({
            type: "dual_frame",
            front: frontNamed ? { timestamp_ms: timestampMs, landmarks: frontNamed } : null,
            side: sideNamed ? { timestamp_ms: timestampMs, landmarks: sideNamed } : null,
          }));
        }
        ```

## 附录: 候选方案（未实现）

### Phase 2.0 候选方案
- ViewState 类封装每个视角的完整状态（包括 buffer）— 当前只分离了 prev_landmarks 和 prev_hip_height

### Phase 2.2 候选融合策略
- 可见度加权平均：对 >= 0.5 可见度的视角按可见度比例加权 — 当前采用取 max 策略

### Phase 2.3 候选优化
- 两路都跑动作识别，取 score 更高的结果 — 当前只用 front 做动作识别

### Phase 4.4 候选方案
- 双摄像头手部识别（HandLandmarker × 2）— 第一版不做，原因：
  1. 热量计算不需要手部（motion_energy 基于 core landmarks）
  2. 性能考虑（双 Pose + 双 Hand 可能导致帧率 < 15fps）
  3. 后端不接收手部数据
  4. 需要额外的左右手匹配和双视角融合逻辑
- 工厂函数 createMediaPipeInstance() — 当前直接在 DisplayView 内部管理，不抽离

### Phase 4.5 性能优化候选
- 错峰推理（front 偶数帧，side 奇数帧）
- 降低推理频率到 15fps
- 降低摄像头分辨率到 960x540

### Phase 5.3 候选方案
- 单摄像头模式调用旧的 sendLandmarks() — 当前统一使用 sendDualFrame()

## 附: 不在本 task 范围内

- 摄像头自动标定（需要 OpenCV stereo calibration）
- 3D 坐标重建（需要坐标系转换和三角测量）
- 关键点级别的融合（需要 3D landmark 融合算法）
- 三摄像头或更多摄像头（架构需要重构）
- 深度信息利用（需要深度摄像头和深度图处理）
- 自动视角判断（需要肩膀连线方向识别）
- 动作识别的完整融合（第一版只做热量计算融合）
